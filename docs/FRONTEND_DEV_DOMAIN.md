# dev.rexten.live и rexten.live — один фронтенд

## Как устроено в репозитории

В `nginx/default.conf` в **одном** блоке `server` указаны три имени:

- `rexten.live`
- `www.rexten.live`
- `dev.rexten.live`

Все три проксируются на **один и тот же** upstream `frontend` (контейнер Next.js на порту **3003** в Docker-сети). Отдельного контейнера или порта «только для dev» в `docker-compose.yml` **нет**.

То есть после **одной** успешной пересборки `frontend` и перезапуска контейнера **оба** домена должны отдавать **одинаковую** вёрстку и JS. Различия «на проде новый hero, на dev старый» при такой схеме **не нормальны** и означают одно из ниже.

## Если dev выглядит по-старому, а prod — нет

1. **DNS**  
   Проверьте, что `dev.rexten.live` и `rexten.live` указывают на **один** сервер (или на один балансировщик с тем же backend). Если dev ведёт на другой IP — там может быть старый PM2, другой compose или старый образ.

2. **Конфиг nginx на сервере не совпадает с репозиторием**  
   Сравните живой конфиг с `nginx/default.conf` из git. Иногда dev вручную проксируют на `localhost:3004` (старый `npm run dev`).

3. **Кэш браузера / CDN**  
   Жёсткое обновление (Ctrl+Shift+R), инкогнито, сброс кэша для `/_next/static` на CDN (если есть).

4. **Фронт не пересобран после `git pull`**  
   Образ `frontend` нужно пересобрать и контейнер пересоздать — см. `scripts/deploy-frontend-docker.sh`.

## Локальная разработка (порт 3004)

`npm run dev` в `frontend/` — это **не** отдельный Docker-сервис для `dev.rexten.live`, а процесс на машине разработчика. Домен `dev.rexten.live` в продакшен-архитектуре репозитория **не привязан** к порту 3004.
