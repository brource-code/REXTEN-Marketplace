# План подробного логирования активности (Rexten)

Документ фиксирует **целевой список событий** для `activity_logs` и связку с `ActivityService`.  
Уже подключено: вход/выход/регистрация, Google-вход, одобрение/отклонение/блок компании, блокировка сотрудника из суперадминки.

**Формат записи:** сегмент (`admin` | `business` | `client` | `system`), категория, действие, `entity_type` + `entity_id`, `company_id`, `description`, `metadata` (компактно).

---

## 1. Бронирования (`category: booking`)

| Событие | Примечание |
|---------|------------|
| Создание брони | источник: клиент / бизнес / публичная запись / повтор |
| Изменение даты, времени, длительности, услуги, специалиста | diff в metadata |
| Смена статуса | new → pending → confirmed → completed / cancelled / no-show |
| Заметки бизнеса / клиента | отдельно или одно событие |
| Перенос | было → стало |
| Отмена | кто отменил, причина |
| Завершение | сумма, при необходимости связь с оплатой |
| Появление связи с заказом | `order_id` |
| Повторяющиеся цепочки | создание / изменение / пауза / отмена цепочки |

---

## 2. Клиенты / CRM (`category: user` или `client`)

| Событие |
|---------|
| Добавление клиента вручную |
| Редактирование ФИО, телефона, email, тегов, источника |
| Объединение дублей (если появится) |
| Блокировка клиента для компании (если появится модель) |

---

## 3. Отзывы (`category: review`)

| Событие |
|---------|
| Новый отзыв (рейтинг, привязка к booking) |
| Ответ бизнеса |
| Редактирование / удаление (модерация суперадмином) |

---

## 4. Услуги и расписание (`service` / `settings`)

| Событие |
|---------|
| CRUD услуги, деактивация |
| Изменение цены, длительности, категории |
| Рабочие часы, исключения, выходные |
| Ручная блокировка слота |

---

## 5. Команда (`company` / `user`)

| Событие |
|---------|
| Приглашение / удаление сотрудника |
| Смена роли / прав |
| Смена владельца компании |

---

## 6. Профиль компании (`company`)

| Событие |
|---------|
| Редактирование названия, slug, описания, контактов, логотипа |
| Подписка / тариф (если актуально) |

*(Одобрение / отклонение / блок компании — уже логируются.)*

## 7. Объявления (`advertisement`)

| Событие |
|---------|

---

## 8. Платежи и заказы (`order` или отдельная категория)

| Событие |
|---------|
| Создание заказа, успешная / неуспешная оплата, возврат |

---

## 9. Клиентский сегмент (`segment: client`)

| Событие |
|---------|
| Бронь с маркетплейса / ЛК |
| Отмена / перенос клиентом |

---

## 10. Суперадмин (`segment: admin`)

| Событие |
|---------|
| Массовые действия по пользователям / компаниям |

## 11. Система (`segment: system`)

| Событие |
|---------|
| Webhooks (Stripe и др.), ошибки доставки |

---

## Рекомендации по внедрению

1. Вызывать `ActivityService` из **сервисного слоя** (например `BookingService`), а не дублировать в 10 контроллерах.
2. В `metadata` — только ключевые поля (`old_status`, `new_status`, `booking_date`, `service_id`), без полных дампов ПДн.
3. Внедрять **фазами**: фаза 1 — брони + отзывы; фаза 2 — CRM + услуги; фаза 3 — объявления и платежи.

## Общая страница активности (уже есть)

- **URL:** `/superadmin/activity-log`
- **Код:** `frontend/src/app/superadmin/activity-log/page.jsx`
- **Таблица / фильтры:** `frontend/src/app/superadmin/activity-log/_components/`

API: `GET /api/admin/activity-log`, статистика: `GET /api/admin/activity-log/stats` (под суперадмин-токеном).
